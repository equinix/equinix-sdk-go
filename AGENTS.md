# AI Agents Guide

This file provides context and instructions for AI agents (GitHub Copilot, OpenAI Codex, and similar tools) working in the `equinix/equinix-sdk-go` repository.

## Repository Overview

This is the official Go SDK for Equinix services. The SDK is generated from OpenAPI specifications for each Equinix service. Each service is maintained as a separate Go package under `services/<service>/`.

## Repository Structure

| Path | Purpose |
|------|---------|
| `Makefile` | Top-level tasks: `generate-all`, `patch-all`, `onboard-service`, `lint`, `test` |
| `Makefile.<service>` | Per-service tasks: `fetch`, `patch`, `generate`, `validate` |
| `spec/services/<service>/oas3.fetched/` | Latest fetched OpenAPI spec for a service |
| `spec/services/<service>/oas3.patched/` | Patched OpenAPI spec ready for code generation |
| `spec/services/<service>/patches/` | Patch files applied to the fetched spec |
| `services/<service>/` | Generated Go client code for a service |
| `templates/services/<service>/` | Custom openapi-generator templates for a service |
| `templates/Makefile.sdk` | Template used to scaffold `Makefile.<service>` |
| `templates/.github/workflows/sync.yaml` | Template used to scaffold sync workflow for a service |
| `patches/services/<service>/` | Post-generation patches applied to generated Go code |
| `script/onboard_service.sh` | Interactive shell script to onboard a new service |
| `config/openapi-generator.json` | Configuration for `openapi-generator` |
| `examples/services/<service>/` | Hand-crafted usage examples |

## Key Concepts

- **Code generation**: Go client code is generated by `openapi-generator` from OpenAPI specs. Do not manually edit files under `services/<service>/` — they will be overwritten on the next sync.
- **Spec patching**: When a fetched spec has issues, create `.patch` files in `spec/services/<service>/patches/` and apply them with `make -f Makefile.<service> patch`.
- **Post-generation patching**: Patches applied to already-generated Go code live in `patches/services/<service>/`.
- **Onboarding**: Use `make onboard-service` or the [Onboard new API](.github/workflows/onboard-api.yaml) GitHub Actions workflow to add support for a new Equinix service.

## Common Tasks for Agents

### Onboard a new API service

Use the [Onboard new API](.github/workflows/onboard-api.yaml) GitHub Actions workflow (preferred), or run:

```bash
make onboard-service
```

When prompted, provide:
1. A lowercase alphanumeric service name (e.g. `mysvcv1`)
2. The URL to the OpenAPI spec (e.g. `https://docs.equinix.com/api-catalog/mysvc/openapi.yaml`)

### Regenerate a service client

```bash
make -f Makefile.<service> generate
```

### Sync a service to the latest OpenAPI spec

Trigger the `Sync <service> API spec` GitHub Actions workflow, or run:

```bash
make -f Makefile.<service> fetch
make -f Makefile.<service> patch
make -f Makefile.<service> generate
```

### Add or update a spec patch

1. Edit files in `spec/services/<service>/oas3.patched/`
2. Generate a patch: `git diff spec/services/<service>/oas3.patched > spec/services/<service>/patches/<index>-<description>.patch`
3. Verify: `make -f Makefile.<service> patch`

### Run tests

```bash
make test
```

### Lint

```bash
make lint
```

## Coding Guidelines

- All Go client code under `services/` is generated — do not edit it directly.
- Follow existing naming conventions: service names are lowercase alphanumeric (e.g. `metalv1`, `fabricv4`).
- Use [Conventional Commits](https://www.conventionalcommits.org/) for commit messages (e.g. `feat:`, `fix:`, `sync:`).
- When adding new custom templates or patches, document why the change is necessary in a comment or PR description.
- The Go module version is tracked in the `version` file and `go.mod`.
